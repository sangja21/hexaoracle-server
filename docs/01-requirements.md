# 📄 01. Requirements — HexaOracle

## 1. 개요
HexaOracle은 사용자가 주역 점괘를 생성하고, AI 해석을 받아볼 수 있는 웹 기반 서비스입니다.  
본 문서는 기능 요구사항(FR), 비기능 요구사항(NFR), 그리고 각 기능의 수용 기준(AC)을 정의합니다.  
요구사항은 다중 인스턴스 환경과 AI 해석 워크로드를 고려하여 설계됩니다.

### 2.2 점괘 생성 API (라인 포맷: 6·7·8·9)

- 사용자가 프론트에서 동전 던지기(척전법)로 **6효**를 확정한다.
- 백엔드로 **라인 값 배열**을 전송한다(각 원소 ∈ {6,7,8,9}, **아래→위** 순서).
- 백엔드는 본괘/변괘/변효를 **재계산**하고, 괘 번호 및 효사 정보를 반환한다.

**AC**
- **입력 형식**: `lines: number[6]` (예: `[6,7,8,9,7,8]`, 아래→위)
- **처리 규칙**:
    - 6·8 = 음(0), 7·9 = 양(1) → 본괘 비트 계산
    - 6·9 위치만 반전 → 변괘 계산
    - 상·하괘 매핑으로 64괘 번호 도출 → `primaryId`, `relatingId`
- **출력**: `primaryId`, `relatingId`, `changingLines`(1~6), (옵션) `dictionary`(괘/효사 텍스트)
- **검증 실패**: 길이≠6 또는 값∉{6,7,8,9} → **400 (Problem+JSON)**
- **멱등성**: 동일 `Idempotency-Key`로 재요청 시 **중복 저장 없음**(동일 응답 반환)

### 2.3 AI 해석 요청 API (Job)
- 생성된 괘와 사용자 질문을 기반으로 AI 해석 제공
- 비동기 Job 처리 구조로 응답 (즉시 완료 or 폴링/웹소켓)
- **AC**:
    - 입력: 괘 번호, 효사, 사용자 질문
    - 출력: AI 해석 텍스트
    - Job 상태: `PENDING`, `IN_PROGRESS`, `COMPLETED`, `FAILED`
    - LLM 타임아웃 시 기본 템플릿 해석 반환

---

### 2.4 히스토리 조회 API
- 사용자별 점괘 생성 및 해석 기록 관리
- 익명 사용자 기록은 토큰 기준으로 저장, 로그인 시 계정과 병합
- **AC**:
    - 페이지네이션 지원
    - 기록에는 날짜, 질문, 괘 번호, 해석 포함
    - 로그인/익명 모두 조회 가능

---

### 2.5 사전(Dictionary) 조회 API
- 각 괘와 효사에 대한 기본 해설, 한자 표기, 영어 번역 제공
- **AC**:
    - 입력: 괘 번호 or 효 번호
    - 출력: 기본 해설, 출처, 번역문
    - DB에 존재하지 않을 경우 404 반환

## 3. 비기능 요구사항 (Non-Functional Requirements)

### 3.1 성능
- 점괘 생성 API 평균 응답 시간: 200ms 이내
- AI 해석 Job은 5초 내 90% 완료 목표

### 3.2 보안
- API Key는 서버 측에서만 보관 (프론트 전송 금지)
- HTTPS 통신 필수
- JWT 기반 인증/인가

### 3.3 데이터 무결성
- 점괘/효사 데이터 변경 시 버전 관리
- 히스토리 데이터는 삭제 전 백업

### 3.4 다중 인스턴스 환경
- 세션/토큰 검증은 stateless 방식
- Job 큐는 Redis 등 외부 스토어 사용

### 3.5 관측성
- API 호출 로깅
- Job 처리 현황 모니터링
- 에러 알림 Slack/Webhook 연동

---

## 4. 수용 기준 (Acceptance Criteria) 요약
- 모든 API는 익명 토큰 또는 로그인 토큰을 필수로 사용
- 잘못된 입력에 대해 명확한 에러 코드/메시지 반환
- AI 해석은 LLM 실패 시에도 기본 해설 제공
- 다중 인스턴스 환경에서 Job 처리와 히스토리 저장이 안정적으로 동작
